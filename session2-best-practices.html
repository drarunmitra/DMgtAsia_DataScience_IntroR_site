Best Practice #3:
Tool Choices

Why R (and not other tools)?


  • Born from S language, specifically designed for statistical analysis
  • The global epidemiology and biostatistics community uses R
  • Hundreds of packages for health research
  • Open-source: you can see how calculations are done
  • Windows, Mac, Linux: runs even on older computers
  • Thousands of tutorials, Stack Overflow answers

R vs. Alternatives for Health Research

Tool Strength Limitation
Excel Accessible, intuitive Prone to errors, not reproducible
SPSS/Stata User-friendly GUI Expensive, less transparent, limited Customization Options
Tableau/Jamovi Less Expensive than others Limited Analysis Options
Python General-purpose, flexible Steeper learning curve for epidemiologists
R Purpose-built for stats, reproducible, free Slightly steeper initial learning curve

The R Ecosystem


  • R Programming Language: The Engine!
  • RStudio IDE: The Dashboard!
  • Quarto/R Markdown: Combine narrative text + code + output (reproducible documents)
  • tidyverse: Collection of R packages designed for modern data workflows

Best Practice #4:
Literate Programming

What is Literate Programming?

Code + Narrative + Output = Single Document

Start with a DATA DICTIONARY

The Concept (Donald Knuth, 1984)

  • Programs should be written for people to read, not just for machines to execute
  • Mix explanation with code
  • Generate analysis reports that are reproducible and transparent

In Practice: Quarto/R Markdown

Write it once. The document weaves narrative, code, and output together:

attack_rates.qmd

# Age-Stratified Attack Rates

Attack rates by age group for the Sholayur Acute Gastroenteritis outbreak.

## Data Preparation

```{r}
age_data <- read_csv(here("data", "family_members.csv")) |>
  mutate(age_group = factor(age_group,
    levels = c("0-4", "5-17", "18-64", "65+")))
```

## Results

```{r}
age_data |>
  group_by(age_group) |>
  summarise(
    n_symptomatic = sum(symptomatic, na.rm = TRUE),
    n_total       = n(),
    attack_rate   = n_symptomatic / n_total * 100
  )
```

Naming Conventions

Naming Conventions

  • Scripts:
    • stroke_data_exploration.R, dengue_data_visualization.R, instead of scrpit1, script2
    • Can consider 01_, 02_ … for indicating sequential execution

Naming Conventions (cont.)

  • Variables:
    • age_group not ageGrp or AgGr (consistency matters)
    • Do not start variable names with numbers
    • Consider adding prefix/suffix to indicate variable type (state_fct, age_group_chr)
    • dem_q1, dem_q2, dem_q3, health_financing_q1, health_financing_q2
    • Add variable labels if necessary using the labelled package

Naming Conventions

  • Files:
    • outbreak_analysis_2025_11_07.R (date in YYYY_MM_DD format)
  • Dates:
    • Always ISO 8601 (YYYY-MM-DD) in code for consistency across regions
    • Use _date suffix for indicating dates
  • Versions:
    • Use version for better management outbreak_analysis_2025_11_07_v2.R instead of outbreak_analysis_2025_11_07_final_final.R

Best Practice #6: Coding Style

Why Coding Style Matters

Code is read 10x more often than it’s written.

The Google R Style Guide Approach

# GOOD: Clear, readable, consistent
calculate_attack_rate <- function(symptomatic_count, total_count) {
  # Calculate attack rate as percentage
  # Args:
  #   symptomatic_count: Number of symptomatic individuals
  #   total_count: Total population at risk
  # Returns:
  #   Attack rate as percentage (0-100)
  
  if (total_count == 0) {
    return(NA)
  }
  
  (symptomatic_count / total_count) * 100
}

# BAD: Unclear naming, no comments
ar <- function(s, t) {
  (s/t)*100
}

Example: Good vs. Bad

  • Good
d<-read.csv('file.csv')
d$ag<-cut(d$a,c(0,5,18,65,150))
d$ar<-ifelse(d$d==1,1,0)*100/d$t

Best Practice #9: Prompting AI Well

AI is Now Part of the Workflow

The use of AI is inevitable. Using it well is a choice.

  • It writes code, drafts text, and explains errors in seconds
  • But it is fluent and confident even when it is wrong
  • What you get back depends on the quality of what you ask for
  • Stay the Human in the Loop: AI assists, you decide

The prompt, not the model, decides whether the output is any good.

Four prompting strategies

Strategy What it is Best for
Zero-shot Just ask, no examples Descriptive stats
Explicit Numbered steps incl. checks Workflows that mustn’t skip steps
Chain-of-thought “Think step by step” Auditing the reasoning
Hybrid Steps + reasoning + format Inferential — the default

Rule of thumb: descriptive → zero-shot; inferential → hybrid.

Example: Vague vs. Briefed

  • Vague
  • Briefed
Analyse this data and run a t-test.

Skips the assumptions, guesses the design, picks a test for you.

You are a biostatistician.
<context> two-arm trial, 20 patients per arm,
outcome is systolic blood pressure </context>
<data> ... </data>

1. Test for normality and equal variance.
2. Choose the correct test and justify it.
3. Report a table: statistic, df, p-value, 95% CI.
4. Interpret in plain language.

Iterate, Do Not Perfect

  • The first answer is the average answer; the second is usually better
  • Refine in small steps rather than crafting one perfect prompt
  • Ask for alternatives and for the reasoning, not just the result
  • Resist sycophancy: ask the model to critique, not to confirm
    • “Grade this analysis against these criteria” beats “this looks fine, right?”

Prompting is Part of Your Methods

The same discipline as Reproducibility (#2) and Literate Programming (#4).

  • Save the prompts that worked, inside the project (prompts/)
  • Record the model, its version, and the date
  • Set a low temperature for analysis you must reproduce
  • Validate every AI-generated number in R before it enters a report
  • Never paste identifiable patient data into a cloud LLM

Problem formulation beats perfect wording. Think first, prompt second.
